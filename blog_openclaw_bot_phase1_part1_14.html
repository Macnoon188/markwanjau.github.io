Bot Creation Phase 1 — Part 1

Getting OpenClaw Running and Connecting the Core Components

Published: Monday, 20 April 2026  ·  Author: Mark Andrew Wanjau  ·  8 min read  ·  Tags: AI · OpenClaw · Telegram · Ollama · Automation

IT support can get repetitive. Password resets, printer jams, misconfigured systems — I've done it all. But one question kept nagging me: what if there was a way to have an assistant that handles the repetitive stuff, guides users, and frees up real humans to focus on the tricky problems?

That's what led me to start OpenClaw. My goal isn't just to build a chatbot. I want to build a real IT support agent that thinks like an engineer, responds like a technician, and grows smarter over time.

This article is Part 1 in a series documenting the journey of building this AI agent from the ground up.

🏗️ The Architecture

Here's the simple flow of what we're building:

👤 User sends message
→
✈️ Telegram interface layer
→
🐾 OpenClaw orchestrator
→
🧠 Ollama local LLM
→
💬 Response back to user
🐾 OpenClaw Gateway The central orchestrator. Connects user requests to AI models and routes everything.
🧠 Ollama Runs the local LLM model that powers the assistant's responses.
✈️ Telegram The user-facing interface where interactions happen.
🐧 Ubuntu VM Hosts everything in a controlled, isolated environment.

Right now, we're focused on getting the bot running reliably — not making it intelligent yet. Infrastructure first. Intelligence second.

🎯 Phase 1 Goal

Deploy a working AI assistant that can respond to Telegram messages. No system prompts. No structured ticketing. Just a bot that works and confirms that the infrastructure is solid.

That's the milestone. Simple, measurable, and meaningful. You can't build intelligence on a shaky foundation — so the foundation comes first.

⚙️ What I Deployed

Here's the Phase 1 deployment checklist:

1. OpenClaw Gateway — Up and running, ready to process requests.
2. Ollama LLM Runtime — Loaded and connected to the gateway.
3. Telegram Bot — Configured and linked to OpenClaw.
4. IT Support Agent File — Template ready to define future behaviour.

And then came the moment of truth. I sent my first Telegram message.

💬 First Successful Response

Telegram · OpenClaw · Phase 1 Test
— First message sent — You: "Hello Jaspa, can you help me troubleshoot?" — Processing through OpenClaw → Ollama — Jaspa: [response received] — Phase 1 milestone confirmed ✓ —

It wasn't perfect. Not even close. But it was alive.

It could receive messages, process them through the stack, and respond through Telegram. That's a milestone moment for any developer working on a bot from scratch — the confirmation that your architecture actually works end to end.

📊 Current Status — End of Phase 1

✅ Working

• Jaspa responding on Telegram

• Ollama model running smoothly

• OpenClaw gateway live

• IT support agent file exists

⏳ Still Missing

• System prompt to define AI behaviour

• Structured ticket format

• Real troubleshooting logic and workflow

That gap between "working" and "missing" is exactly where Part 2 comes in.

🔭 What's Next — Phase 1 Part 2

In Bot Creation Phase 1 — Part 2, the focus shifts from infrastructure to intelligence:

• Adding system prompts so Jaspa thinks like an IT engineer
• Structuring ticket outputs for real support workflows
• Teaching the bot to respond intelligently to real support scenarios

Once that's done, Phase 1 will be complete and the bot will move considerably closer to being a functional IT assistant — not just a bot that's alive, but one that's useful.

💭 Final Thoughts

Building this isn't just about code. It's about seeing the pieces connect. Every step in this phase confirmed that the architecture works and the foundation is solid.

That confidence — knowing your stack is real before you try to make it smart — is something you only get by building, not by reading.

Next up: making Jaspa smart enough to handle real IT problems, while still learning from every interaction.
