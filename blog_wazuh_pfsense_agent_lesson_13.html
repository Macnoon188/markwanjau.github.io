Why I Stopped Using the Wazuh Agent on pfSense

And what that mistake taught me about real security architecture

Published: Monday, 13 April 2026  ·  Author: Mark Andrew Wanjau  ·  Tags: Wazuh · pfSense · SIEM · SOC · Architecture · Homelab

At the beginning of building my homelab SOC, my mindset was simple:

"If I want visibility, install an agent."

So when I deployed Wazuh 4.5 inside my Proxmox lab and started monitoring my infrastructure, I installed the Wazuh agent on almost everything I could — including pfSense.

Technically? It worked.
Strategically? It was the wrong move.

And that realization changed how I think about security architecture.

🏗️ The Setup

My lab at the time of this decision included:

• Proxmox as the hypervisor
• VLAN segmentation for network isolation
• Ubuntu Server running Wazuh Manager
• Multiple monitored endpoints across the lab
• pfSense as the firewall and network gateway

At first, installing the Wazuh agent on pfSense felt like the "complete visibility" approach. More logs equals more security — right?

Not exactly.

⚠️ The Problem With an Agent on pfSense

pfSense is a specific kind of system. It is:

• A firewall appliance — not a general-purpose server
• Built on FreeBSD — not a standard Linux environment
• Designed to be lean and hardened
• Sitting at the edge of your network, handling all traffic

Adding a full endpoint agent to a system like this introduces additional resource usage, upgrade complications, potential instability, and — critically — an expanded attack surface on the device that is supposed to be your first line of defence.

But beyond the technical concerns, there was a bigger issue. It simply is not how real-world SOC teams monitor firewalls.

In production environments, firewalls are monitored through:

• Syslog forwarding
• API integrations
• NetFlow / sFlow
• Vendor-native telemetry

Not endpoint agents. That was my first real architecture lesson.

"Just because something can run an agent doesn't mean it should."

🔄 The Shift — Moving to Syslog Ingestion

Instead of running the Wazuh agent on pfSense, I made four deliberate changes:

01

Removed the agent completely from pfSense. Clean slate.

02

Configured pfSense to forward logs via Syslog to my Wazuh manager's IP on UDP 514.

03

Tuned Wazuh to properly ingest and parse firewall logs — filtering noise, highlighting meaningful events.

04

Shifted focus from log quantity to alert quality. Fewer, better signals.

The difference was immediate.

❌ Agent on pfSense

Extra resource load on the firewall

FreeBSD compatibility friction

More attack surface at the edge

Not aligned with enterprise practice

Noisy, unfiltered log flood

✅ Syslog Forwarding

Firewall stays lean and hardened

Wazuh parses logs centrally

Smaller attack surface

Mirrors production SOC architecture

Cleaner, higher-quality alerts

📐 What This Taught Me

This wasn't just a configuration change. It was a mindset shift.

1 — Security Is Not About Installing More Tools

It's about understanding where each tool belongs. Just because something can run an agent doesn't mean it should. Tool placement is an architectural decision, not a default.

2 — Architecture Matters More Than Activity

A clean, intentional architecture is stronger than an over-engineered one. Security maturity is often knowing when to simplify — not when to add more.

3 — Real-World Alignment Is Non-Negotiable

My goal is not to build a flashy homelab. It is to prepare for real SOC and infrastructure environments. In real environments: firewalls send logs, SIEMs ingest them, analysts investigate them, and appliances stay hardened. That separation matters.

🎯 The Bigger Lesson

This experience forced me to stop thinking like:

"How do I make this work?"

And start thinking like:

"Is this the right way to design this?"

That shift — from execution to architecture — is something I didn't expect to learn this early in my journey.

But I'm glad I did.

🔭 What's Next in the Lab

Now that firewall monitoring is properly structured, I'm focusing on:

• Log tuning and noise reduction
• Writing proper SOC-style incident reports
• Simulating lateral movement scenarios
• Expanding into Windows-based logging
• Strengthening detection engineering fundamentals

The lab is slowly becoming less about experimenting — and more about building intentionally.

To what, you may ask?

I guess we'll just wait and see 😊

If you're building your own homelab and you've installed an agent everywhere "just because you can" — I get it. I did the same.

But sometimes growth in cybersecurity isn't about adding more. It's about removing the unnecessary. And documenting what you learned along the way.
