📖 Part 3 of Series · Building Hermes

From AI Assistant to AI Engineering Partner

How I Stopped Prompting and Started Building a System

Published: Monday, 11 May 2026  ·  Author: Mark Andrew Wanjau  ·  10 min read  ·  Tags: AI · Hermes · Engineering · CryptXSnare · Automation

When I first started building Hermes, my goal was simple.

Can I build an AI assistant that helps manage my homelab?

Like many people experimenting with AI, I imagined a chatbot that could answer questions, execute terminal commands, and automate repetitive tasks. It did all of those things.

But after several weeks of building, deploying, breaking, rebuilding, and integrating Hermes into my cybersecurity lab, I realized something.

I wasn't looking for another chatbot.

I wanted an engineering partner.

That realization completely changed the direction of the project.

The Turning Point

The change didn't happen because of a breakthrough model or a new API.

It happened because of failure.

While developing CryptXSnare — my SOC platform — Hermes helped implement a new incident triage workflow. The feature itself wasn't difficult. The aftermath was.

Multiple rapid patches eventually corrupted a React component. The project stopped compiling. Instead of continuing to add features, we had to stop everything and ask a different question.

Why did this happen?

The answer wasn't "because AI made a mistake." The answer was much more interesting. Hermes had no engineering process. It knew how to edit files. It didn't know how to engineer software.

The Difference

Most AI coding assistants work like this — and it does work. But it doesn't scale.

The difference isn't more intelligence. It's more discipline.

Engineering Before Coding

One of the first lessons Hermes learned was that coding should never be the first step. Instead, every repository should go through an Engineering Loop. This became the foundation of Hermes-Core.

🔍 Understand // read context
↓
🧪 Audit // check health
↓
📋 Plan // define approach
↓
⚙️ Execute // implement
↓
✅ Verify // compile & test
↓
📄 Document // record learning
↓
💾 Commit & Push // git
↓
🧠 Update Knowledge // Hermes-Core

Every project now follows the same workflow. No shortcuts.

The Dependency Audit Lesson

While working on CryptXSnare, Hermes attempted to start the application. The result was immediate.

cryptXsnare — npm run dev
$ npm run dev vite: not found

The problem wasn't Vite. The dependencies had never been installed. Once that was corrected, another issue appeared.

// Dependency Conflict Discovered
react-day-picker → requires → date-fns 2.x or 3.x
project had → installed → date-fns 4.x ← incompatible

The environment wasn't healthy. Instead of forcing the installation, we investigated the dependency graph, corrected the versions, rebuilt the project, and only then continued. That experience inspired a new Hermes capability.

📦 Repository Cloned
↓
🔎 Dependency Audit
↓
🧪 Environment Verification
↓
🏗️ Build
↓
✅ Testing
↓
💻 Development Begins

No more coding inside a broken environment.

When the Build Finally Failed — and That Was Good News

Once the dependencies were healthy, the compiler finally exposed the real issue.

cryptXsnare — TypeScript Compiler
ERROR Threats.tsx Adjacent JSX elements must be wrapped in an enclosing tag.

This wasn't a package issue. It wasn't Node. It wasn't React. It was source code integrity.

Ironically, this was good news.

The infrastructure was finally healthy enough to tell us where the real problem lived.

Hermes-Core Begins to Grow

At this point Hermes had become more than an assistant. It had started accumulating operational knowledge. But another realization followed quickly.

Engineering knowledge didn't quite fit into Skills or Playbooks. So the framework began evolving.

Hermes-Core/ ├── skills/ ├── playbooks/ ├── policies/ ├── memories/ ├── templates/ └── knowledge/     ├── architecture/     ├── decisions/     ├── lessons/     └── postmortems/

Now Hermes wasn't just remembering procedures. It was remembering engineering decisions. Every solved problem became reusable. Every lesson reduced future mistakes.

A New Engineering Policy

The JSX corruption exposed another weakness. Hermes had been applying multiple patches without verifying each one. That changed immediately. A new policy was born.

// One Edit, One Compile — Always

→

Read the entire component

→

Understand the full structure

→

Apply ONE logical edit

→

Compile and verify

→

Repeat

Never perform multiple structural edits without a successful compilation between them. This single rule would have prevented the entire issue.

"Artificial intelligence doesn't replace engineering discipline. It amplifies it."

Project Health Instead of Guesswork

Before writing code, I wanted Hermes to answer one simple question: "Is this project healthy?" That led to the design of a new command.

hermes — /health
$ /health   Project Health Report ───────────────────────────────── Repository ✓ Git Clean   Environment ✓ Node ✓ npm   Dependencies ✓ Installed ✓ Compatible   Build ✗ Failing   Current Error Threats.tsx → Line 416   Recommendation Repair JSX before feature development.

Instead of discovering problems halfway through development, Hermes would identify them before writing a single line of code.

Looking Ahead

Today Hermes helps me develop applications, manage infrastructure, document procedures, and automate repetitive engineering tasks.

Tomorrow, I want it to become an engineering partner capable of maintaining project health, preserving institutional knowledge, and continuously improving the way I build software.

The real goal was never to build an AI assistant.

The goal was to build a system that learns alongside me. And I think we're finally heading in that direction.

Final Thoughts

One lesson stands out more than any other from this entire journey.

Artificial intelligence doesn't replace engineering discipline. It amplifies it.

The better the process, the better the outcome. That's why the future of AI-assisted development isn't just about better models. It's about building better systems around them.

Hermes is no longer just an assistant. It's becoming an engineering operating system.
